## 2.4 Persistent Level Actors
The Persistent Level contains only the following actors. Everything else is World Partition cell content:
- World Partition Grid
- Game Mode (BP_MV_GameMode)
- Game State (BP_MV_GameState)
- Player Start markers
- Global Manager actors (spawned via GameMode; see Section 3)
- Post-process volume (global; see Rendering Bible companion doc)
- Sky Atmosphere + Sky Light
- Directional Light (sun; controlled by TimeManager)

⚠ Design Rule
Do not place gameplay actors directly in the Persistent Level. All gameplay content belongs in World Partition cells. Persistent Level actors are infrastructure only.

## 5.4 Interaction System (UMV_InteractionComponent)
The interaction system uses a sphere trace (radius 200cm) from player capsule center every 0.1 seconds (10Hz). It collects all actors implementing IMV_Interactable, sorts by dot product to player forward vector (facing priority), and presents the top result as the active interaction prompt.
### Interaction Priority Order
- Quest-critical object (overrides all others)
- NPC (social interaction)
- Farming interactable (crop, water source)
- World object (chest, tool, sign)
- Resource node (forage, mine, fish spot)
- Environmental detail (lore object, examine)
### IMV_Interactable Interface
class IMV_Interactable
{
    // Returns the interaction prompt text
    virtual FText GetInteractionPrompt() const = 0;

    // Returns interaction priority (higher = prioritized)
    virtual int32 GetInteractionPriority() const = 0;

    // Called when player triggers interaction
    virtual void OnInteract(AMV_PlayerCharacter* Interactor) = 0;

    // Returns true if currently interactable
    // (may depend on quest state, knack level, etc.)
    virtual bool CanInteract(AMV_PlayerCharacter* Interactor) const = 0;
}

# Section 6 — NPC Systems
## 6.1 NPC Architecture Overview
Mooseville NPCs do not use Behavior Trees or standard UE5 AI. They use a custom Schedule System — a time-driven goal execution framework that makes NPCs feel like they have real lives rather than AI routines.
### NPC Class Hierarchy
ACharacter
  └── AMV_NPCBase (C++)
        ├── AMV_NPC_Named (C++)   — for all story/relationship NPCs
        └── AMV_NPC_Wildlife (C++) — for bears, eagles, deer
## 6.2 Schedule System (UMV_NPCScheduleManagerSubsystem)
Architecture: Each Named NPC has a data asset (DA_NPC_[Name]_Schedule) containing a sorted array of FMV_ScheduleEntry structs. The Schedule Manager subscribes to MV.Time.DayPhaseChanged and evaluates each NPC's active schedule entry.
### FMV_ScheduleEntry Struct
struct FMV_ScheduleEntry
{
    float StartTime;             // Game time (0.0-24.0)
    float EndTime;               // Game time (0.0-24.0)
    FVector TargetLocation;      // World location of destination
    FName ActivityTag;           // e.g., "Farming", "Reading", "Walking", "Eating"
    TArray<FName> DialoguePool;  // Dialogue line IDs available during this activity
    bool bRequiresRelationshipLevel; // If true, only shown if Closeness >= MinCloseness
    int32 MinCloseness;          // Minimum closeness to observe this entry
    FName WeatherOverride;       // If set, NPC moves indoors during rain regardless of schedule
}

## 6.4 Dialogue System
Mooseville uses a custom lightweight dialogue system built on UE5 DataTables and a Dialogue Component. No third-party tool is required.
### FMV_DialogueLine Struct
struct FMV_DialogueLine
{
    FName LineID;
    FText DialogueText;
    FName SpeakerID;
    TArray<FName> RequiredQuestStates;  // Empty = always available
    int32 RequiredClosenessLevel;
    bool bOneShot;                      // Removed from pool after delivery
    bool bRememberedFact;               // Adds a fact to RelationshipData
    FName FactTag;                      // Tag of the fact being remembered
    TArray<FName> PlayerChoiceIDs;      // Empty = no player choice
    float PriorityWeight;               // Higher = selected more often
}
### Dialogue Selection Algorithm
- Filter all lines for NPC by: QuestState conditions, Closeness level, and OneShot not-yet-seen
- Sort remaining by PriorityWeight (descending)
- Select top 3 as candidates; weighted random from top 3
- Check if chosen line references a RememberedFact — if player has shared that fact, boost by ×2 weight

Design Note — No Traditional Dialogue Trees
Mooseville does not use branching dialogue trees with multiple player responses in the traditional sense. Players have 1–3 response choices maximum, and these affect Closeness Points awarded — not story branches. Story pivots happen via Quest completion and Relationship Level, not individual dialogue choices.


# Section 7 — Quest System
## 7.1 Quest Manager (UMV_QuestManagerSubsystem)
Quests in Mooseville are fully data-driven. Every quest is defined in a DataTable (DT_MV_Quests) and executed via a state machine managed by the QuestManager.
### FMV_QuestData Struct
struct FMV_QuestData
{
    FName QuestID;
    FText QuestTitle;
    FText QuestDescription;              // Journal-voice description (player character's voice)
    EQuestType QuestType;                // MainStory, NPCStory, Community, Discovery, Seasonal, Archive
    FName GiverNPCID;                    // NPC who gives/owns this quest (empty for Discovery)
    TArray<FMV_QuestStep> Steps;
    TArray<FName> PrerequisiteQuestIDs; // Must be completed first
    int32 RequiredClosenessLevel;        // Closeness with GiverNPC required to receive quest
    int32 RequiredActState;             // 0 = any, 1 = Act 1+, 2 = Act 2+, 3 = Act 3 only
    bool bTimeLimited;                   // Seasonal quests
    ESeason SeasonAvailable;
    TArray<FMV_QuestReward> Rewards;
}

struct FMV_QuestStep
{
    FName StepID;
    FText StepDescription;
    EStepType StepType;          // GoTo, Collect, Interact, Talk, Craft, Wait
    FName TargetID;              // Location/Item/NPC/Recipe ID depending on StepType
    int32 RequiredCount;         // For Collect steps
    bool bOptional;
    bool bHidden;                // Hidden steps don't appear in journal until triggered
}

struct FMV_QuestReward
{
    ERewardType RewardType;      // Mucks, Item, RelationshipPoints, KnackPoints, JournalEntry
    FName RewardID;
    int32 RewardAmount;
}
## 7.2 Quest State Machine
Each active quest instance runs as a state machine with the following states:
NotStarted → Active → StepComplete (loop) → Completed

Design Note — No Fail States
Quests in Mooseville have
no fail state
. Steps wait indefinitely. Seasonal quests "pause" at season end and resume on the next occurrence of the available season.

## 7.3 Quest Journal Integration
Every quest state change fires the appropriate message channel, which the JournalManager receives and auto-writes a journal entry in the player character's voice. Journal entries are templated with variable substitution:
Template: "I spoke with {NPCNAME} today near {LOCATION}. They asked me to {TASK}.
I've been thinking about it since — {PLAYER_REFLECTION}."

Variables filled from QuestData and RelationshipData at time of writing.

### Crop Growth State Machine (per crop plot tile)
Empty → Seeded → Growing (Days 1 to DaysToGrow) → Harvestable → Wilted (if not harvested within +3 days)
### Plot Tile Actor (BP_MV_CropPlot)
- 1m × 1m tile actor placed in world
- Stores: current CropID, GrowthDay, WateredToday, MooseWaterApplied
- Subscribes to MV.Time.DayPhaseChanged (Dawn); advances growth counter at dawn if watered
- Visual mesh swaps through 4 growth stages via StaticMesh component swap
## 8.2 Fishing System (UMV_FishingComponent)

Replaces the custom UMV_QuestManagerSubsystem quest state machine and the DA_MV_Dialogue system if adopted early in production. Not recommended to adopt mid-production if the custom system is already built.
Integration Notes
Best adopted at project start. If adopted, wire Narrative's quest completion delegates to the GameplayMessageSubsystem channels defined in Section 3.2 to preserve the event bus architecture. Narrative's dialogue system will need custom extensions to support the Remembered Facts mechanic (Section 6.3). Estimated integration: 1 week setup + narrative designer training.

A comprehensive, production-proven dynamic sky, weather, and atmosphere system for UE5. Provides volumetric clouds, animated precipitation (rain, snow), dynamic fog, and time-of-day sky transitions fully compatible with Lumen. Dramatically reduces custom sky and weather shader work.

Configure UDS's time-of-day system to be driven by UMV_TimeManagerSubsystem's float time output — do not let UDS own the clock. Snow coverage can complement the MPC_SeasonBlend.GroundSnowAmount parameter in Winter.

Purpose
Extends UE5's base USaveGame system with slot management, compression, async save/load operations, and save-data migration utilities. Particularly valuable for handling save versioning as Mooseville ships patches and expansions — prevents save file corruption on updates.
Replaces or Enhances
Enhances (does not replace) the UMV_SaveManagerSubsystem described in Section 9. The save manager retains its data architecture; Easy Multi Save handles the serialization pipeline, slot UI patterns, and async I/O.
Integration Notes
Replace the UGameplayStatics::SaveGameToSlot calls in UMV_SaveManagerSubsystem with Easy Multi Save's async equivalents to avoid any main-thread stall during autosave. Implement the plugin's versioning interface on UMV_SaveGame from day one — retroactively adding versioning after launch is painful. The 500ms save write target (Section 13.1 Gate 5) is more reliably achieved with async I/O.

## Plugin 5 — PCG Extended / PCG Biomes

Field
Detail
Plugin Name
PCG Extended (community) or PCG Biomes Framework (Fab)
Source
GitHub (PCG Extended, open-source) or Fab (various PCG Biome packs)
Purpose
Extends UE5's built-in PCG (Procedural Content Generation) framework with additional nodes for biome-blending, density painting, spline-following scatter, and slope-aware placement. Replaces or Enhances

Integration Notes
Evaluate against UE5.4's built-in PCG node library first — Epic has expanded PCG considerably. Only adopt if the built-in nodes are missing key features your Tech Art team needs. All PCG-placed foliage must use ISM (Instanced Static Mesh) components to stay within the foliage draw call budget (50–80 DC; Section 13.2). Coordinate with Data Layer setup so seasonal foliage PCG graphs respect DL_Season_* layer visibility.
